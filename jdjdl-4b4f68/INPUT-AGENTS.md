## Chef Attribute Precedence and Runtime Source of Truth

When analyzing Chef cookbooks, always resolve attribute precedence correctly. `solo.json` (the Chef Solo node JSON) overrides `attributes/default.rb` at runtime and is the authoritative source for variable values. Specifically:
- Document roots are `/var/www/{test.cluster.local,ci.cluster.local,status.cluster.local}` (from `solo.json`), NOT `/opt/server/{test,ci,status}` (from `attributes/default.rb`).
- Treat any value in `solo.json` as the effective runtime value. Attribute defaults in cookbook files that are overridden by `solo.json` are dead code and must not be carried forward into Ansible `group_vars`.
- Flag any discrepancy between `attributes/default.rb` and `solo.json` values in the migration specification.

## Security Hardening Resource Inventory

Enumerate all security hardening actions applied across cookbooks for complete migration coverage:
- **SSH hardening**: `PermitRootLogin no` and `PasswordAuthentication no` applied via `sed` commands with idempotency guards in `security.rb`.
- **UFW firewall**: default deny, allow SSH/HTTP/HTTPS — implemented via `execute` resources with shell idempotency guards.
- **fail2ban**: jails for `sshd`, `nginx-http-auth`, `nginx-limit-req`, `nginx-botsearch`; ban time 3600s, max retry 3.
- **sysctl hardening**: 15 kernel parameters written to `/etc/sysctl.d/99-security.conf` (IP spoofing protection, ICMP redirect rejection, source routing disabled, martian logging, SYN flood protection via `tcp_syncookies`, IPv6 disabled).
- **SSL/TLS**: TLS 1.2/1.3 only, strong cipher suites, `server_tokens off`, rate-limiting zones in `conf.d/security.conf`.
- **Per-site security headers**: HSTS (`max-age=31536000; includeSubDomains`), `X-Frame-Options DENY`, `X-Content-Type-Options nosniff`, `X-XSS-Protection`, `Content-Security-Policy`, `Referrer-Policy`.
- **SSL private key permissions**: `0640` with `root:ssl-cert` ownership — must be preserved exactly.
- **FastAPI `.env` file**: currently `0644` owned by root — flag as a security defect; must be corrected to `0600` owned by the service account.

## FastAPI Service Security Defect Identification

The `fastapi-tutorial` cookbook configures the systemd unit with `User=root`. This is a security defect that must be flagged in the migration specification and corrected during migration — do not replicate it. The migration specification must include:
- Creation of a dedicated `fastapi` system user.
- The systemd unit must run under this dedicated user account.
- The application directory `/opt/fastapi-tutorial` and the `.env` file must be owned by the `fastapi` user.
- The `.env` file must use `mode: '0600'` (not `0644` as currently set).
